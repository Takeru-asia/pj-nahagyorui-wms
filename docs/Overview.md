## 1. Background / Context

那覇魚類は現在、以下のような業務環境で受発注業務を行っている:

- **レガシーな販売管理ソフト**
- **Excel**
- **紙帳票**
- **口頭伝達**

### 組織構成

- **営業部**: 各々が顧客担当者として、顧客からの注文受付から配送まで一貫して担当
- **事務部**: 伝票作成や事務処理を担当

### 制約事項

- 既存のレガシーシステムは回収が不可能

## 2. Opportunity / Problem

### 2.1 在庫管理がほぼできていないため発生している問題

#### 在庫数量の不透明性
- **現在庫数量が把握できない**
  - リアルタイムでの在庫状況が分からない
  - 在庫不足や過剰在庫の判断ができない

#### 棚卸し業務の非効率性
- **棚卸しを高頻度で複数回行わなければならない**
  - 在庫管理ができていないため、頻繁な確認作業が必要
  - 人的コストと時間コストが膨大

#### 先入先出の未実施
- **先入先出(FIFO)ができていない**
  - 古い在庫が残り、品質劣化のリスク
  - 鮮度管理が重要な魚類商品において致命的

#### 原価管理の欠如
- **売上と仕入れが紐づけられず、原価が把握できていない**
  - 適切な利益計算ができない
  - 経営判断に必要なデータが不足

#### 価格管理の不透明性
- **営業部が値上げをせずに原価が上がった品物を、原価が上がる前のロットだと言うことにして販売することができる状況**
  - 利益率の低下
  - 不透明な価格設定による経営リスク
  - ロット管理ができていないことによる問題

### 2.2 販売管理システムがレガシーなために発生している問題

#### 部門間コミュニケーションエラー
- **営業部と事務部の間で伝達ミスが発生**
  - 口頭伝達や紙帳票による情報共有の限界
  - 受注内容の認識齟齬

#### データ入力ミス
- **事務部の打ち間違えが発生**
  - 商品コードを手動で打ち込む方式
  - 受注した本人以外の事務員が入力するため、ミスに気づきにくい
  - 誤発注や誤出荷のリスク

## 3. Goals / Objectives

### 主目標
**在庫管理システムを導入する**

### 期待される効果

1. **在庫の可視化**
   - リアルタイムでの在庫数量把握
   - 在庫状況に基づいた迅速な意思決定

2. **業務効率の向上**
   - 棚卸し作業の削減
   - 先入先出(FIFO)の実現
   - 部門間連携の改善

3. **原価管理の実現**
   - 売上と仕入れの紐付け
   - ロット単位での原価追跡
   - 適正な価格設定の実現

4. **ヒューマンエラーの削減**
   - 入力ミスの防止
   - 伝達ミスの削減

## 5. 主要機能 (想定)

*詳細は要件定義フェーズで確定*

1. **入庫管理**
   - 仕入れ情報の登録
   - ロット管理
   - 受入検品

2. **出庫管理**
   - 出荷指示
   - ピッキング
   - 出荷実績登録

3. **在庫照会**
   - リアルタイム在庫数量表示
   - ロット別在庫
   - 在庫推移

4. **棚卸し機能**
   - 棚卸し計画
   - 実地棚卸し
   - 差異調整

5. **先入先出管理**
   - ロット別在庫管理
   - 出荷順序の自動提案

6. **原価管理**
   - 仕入れと売上の紐付け
   - ロット別原価追跡

## 7. リスクと制約

### リスク

1. **ユーザーの習熟度**
   - レガシーシステムからの移行への抵抗
   - 新システムへの適応に時間がかかる可能性

2. **既存システムとの連携**
   - レガシー販売管理システムとの連携難易度

3. **データ移行**
   - 既存データの整合性確保
   - 初期在庫データの正確性

### 制約

1. **既存システムは回収不可**
   - 既存システムからのデータ取得が困難な可能性

2. **販売管理システムは刷新対象外**
   - 在庫管理システムのみに範囲を限定
